## 行为准则

我们希望所有参与者都能在一个友好、包容的环境中进行交流与合作。请尊重其他贡献者，保持专业和建设性的讨论态度。

## 如何贡献

### 报告问题

如果您发现了 bug 或有改进建议，请通过 GitHub Issues 提交问题报告。提交问题时，请尽可能详细地描述：

1. 问题的具体表现
2. 复现步骤
3. 预期行为与实际行为的差异
4. 运行环境（操作系统、浏览器版本等）
5. 如可能，附上截图或错误日志

### 提交功能请求

如果您希望系统添加新功能，请通过 Issues 提交功能请求，并描述：

1. 您希望添加的功能
2. 该功能将解决什么问题或满足什么需求
3. 可能的实现方式（如有想法）

### 提交代码

如果您希望直接贡献代码，请按照以下步骤操作：

1. Fork 本仓库
2. 创建您的特性分支 (`git checkout -b feature/amazing-feature`)
3. 提交您的更改 (`git commit -m 'Add some amazing feature'`)
4. 推送到分支 (`git push origin feature/amazing-feature`)
5. 创建一个 Pull Request

## 开发环境设置

要设置本地开发环境，请按照以下步骤操作：

1. 确保您已安装 Node.js（推荐版本 v14 或更高）
2. 克隆仓库：`git clone https://github.com/muzig/markdown-review-web.git`
3. 进入项目目录：`cd markdown-review-web`
4. 安装依赖：`npm install` 或 `pnpm install`
5. 复制环境变量文件：`cp .env.example .env`
6. 根据需要修改 `.env` 文件中的配置
7. 启动开发服务器：`npm start`

## 代码规范

为保持代码质量和一致性，请遵循以下规范：

### JavaScript/Node.js

- 使用 ES6+ 语法
- 使用 2 空格缩进
- 使用分号结束语句
- 变量和函数名使用驼峰命名法（camelCase）
- 类名使用帕斯卡命名法（PascalCase）
- 常量使用全大写下划线命名法（UPPER_SNAKE_CASE）

### 前端代码

- 遵循 Vue.js 官方风格指南
- 组件名使用多词组合，避免与 HTML 元素冲突
- Props 定义应尽可能详细，包括类型和默认值
- 避免在模板中使用复杂表达式，复杂逻辑应移至计算属性或方法

### 注释

- 为函数、类和复杂逻辑添加清晰的注释
- API 端点应有注释说明其功能、参数和返回值
- 使用 JSDoc 风格的注释

## 提交流程

1. 在开始工作前，请先创建 Issue 描述您计划解决的问题或添加的功能
2. 创建对应的分支进行开发
3. 完成开发后，确保代码通过所有测试
4. 提交 Pull Request，并在描述中引用相关 Issue
5. 等待代码审查，并根据反馈进行必要的修改
6. 一旦 PR 被批准，它将被合并到主分支

### 提交信息规范

提交信息应遵循以下格式：

```
<类型>(<范围>): <简短描述>

<详细描述>

<引用相关 Issue>
```

类型包括：

- feat: 新功能
- fix: 修复 bug
- docs: 文档更新
- style: 代码风格调整（不影响功能）
- refactor: 代码重构
- perf: 性能优化
- test: 添加或修改测试
- chore: 构建过程或辅助工具的变动

示例：

```
feat(review): 添加自定义复习间隔设置

允许用户为不同类型的文档设置不同的复习间隔，
增强了系统的灵活性和个性化程度。

Closes #42
```

## 分支策略

- `main`: 主分支，包含稳定版本的代码
- `develop`: 开发分支，包含最新的开发代码
- `feature/*`: 功能分支，用于开发新功能
- `bugfix/*`: 修复分支，用于修复 bug
- `release/*`: 发布分支，用于准备新版本发布

## 版本发布流程

我们使用语义化版本控制（[SemVer](https://semver.org/lang/zh-CN/)）来管理版本号：

- 主版本号（X.0.0）：不兼容的 API 变更
- 次版本号（0.X.0）：向下兼容的功能性新增
- 修订号（0.0.X）：向下兼容的问题修正

发布新版本的流程：

1. 从 `develop` 分支创建 `release/vX.Y.Z` 分支
2. 在发布分支上进行最终测试和修复
3. 将发布分支合并到 `main` 和 `develop`
4. 在 `main` 分支上创建版本标签
5. 更新 CHANGELOG.md 文件
6. 发布新版本
